# Instrument Server

BoltConn's instrument server is a WebSocket service used by two rule actions:

- `.INSTRUMENT` publishes formatted telemetry/debug strings.
- `.REQUEST` sends a connection snapshot to a client and waits for a route decision.

The server starts only when the top-level `instrument:` config block is present. The internal
message bus still runs without it, but no remote client can subscribe.

## Configuration

Enable the service in `config.yml`:

```yaml
instrument:
  api_addr: 9001
  secret: instrument-secret
  cors_allowed_list:
    - "http://localhost:3000"
```

**Fields:**

| Field | Required | Description |
|---|---|---|
| `api_addr` | yes | Port number or socket address to bind |
| `secret` | no | Shared secret required in the WebSocket query string |
| `cors_allowed_list` | no | Extra browser origins allowed to open the WebSocket |

## WebSocket Endpoint

**Path:** `GET /subscribe`

**Query parameters:**

| Parameter | Required | Description |
|---|---|---|
| `id` | yes | One or more numeric subscriber IDs, comma-separated |
| `secret` | when configured | Must match `instrument.secret` |

**Examples:**

```text
ws://127.0.0.1:9001/subscribe?id=101
ws://127.0.0.1:9001/subscribe?id=101,200&secret=instrument-secret
```

**Connection rules:**

- Each subscriber `id` can have only one live WebSocket subscriber at a time.
- A single WebSocket may claim multiple IDs by passing `id=1,2,3`.
- If any requested ID is already in use, the handshake fails with `409 Conflict`.
- Missing or invalid `id` returns `400 Bad Request`.
- Wrong `secret` returns `401 Unauthorized`.
- Unlike the REST API, the instrument WebSocket does not use a bearer token header; authentication
  is only the `secret` query parameter.

### Browser Origin Checks

If the request includes an `Origin` header, BoltConn accepts:

- `http://localhost...`
- `http://127.0.0.1...`
- `https://localhost...`
- `https://127.0.0.1...`
- `file://...`
- Anything explicitly listed in `cors_allowed_list`

Non-browser clients usually omit `Origin`, so this check normally does not apply to them.

## Wire Format

Server-to-client messages are always text frames in this envelope:

```text
<id>:<message>
```

`id` is the subscriber ID. `message` depends on the rule action:

- `.INSTRUMENT`: plain formatted text
- `.REQUEST`: JSON serialized from `boltapi::instrument::RequestPayload`

Client-to-server messages are only meaningful for `.REQUEST` responses and must be plain JSON
text frames:

```json
{
  "sub_id": 200,
  "request_id": 42,
  "route": "FALLBACK"
}
```

The server ignores a response when `sub_id` does not match the subscriber that owns
`request_id`.

## `.INSTRUMENT`

Rule example:

```yaml
- .INSTRUMENT:
    id: 101
    matches: PROCESS-NAME, curl
    message: "[{time.hms_ms}] {process.name} <- {process.parents.0.name} -> {addr.dst}"
```

When the rule matches, BoltConn formats `message` and publishes it to subscriber `101`.
Templates are validated when the rule set is built, so unknown placeholders fail config load or
reload instead of failing at runtime.

### Template Variables

**Address and inbound:**

- `addr.src`: source socket address
- `addr.dst`: original destination
- `addr.resolved_dst`: resolved destination, or `N/A`
- `ip.local`: local IP selected for the connection, or `N/A`
- `inbound.type`: `tun`, `http`, or `socks5`
- `inbound.port`: inbound listening port, or `N/A`
- `inbound.user`: inbound username, or `N/A`
- `conn.type`: `tcp` or `udp`

**Process:**

- `process.name`
- `process.cmdline`
- `process.path`
- `process.cwd`
- `process.pid`
- `process.tag`
- `process.parent.pid`
- `process.parent.name`
- `process.parent.path`
- `process.parent.cmdline`
- `process.parent.cwd`
- `process.parent.all.json`
- `process.parents.<n>.pid`
- `process.parents.<n>.name`
- `process.parents.<n>.path`
- `process.parents.<n>.cmdline`
- `process.parents.<n>.cwd`

**Time:**

- `time.hms_ms`: `HH:MM:SS.mmm`
- `time.datetime`: `YYYY-MM-DD HH:MM:SS`
- `time.datetime_ms`: `YYYY-MM-DD HH:MM:SS.mmm`
- `time.rfc3389`: placeholder name used by code; value is an RFC 3339 timestamp

### Template Behavior

- Missing scalar values render as `N/A`.
- `process.parent.all.json` renders `[]` when no process metadata is available.
- `process.parent.*` is the same immediate ancestor as `process.parents.0.*`.
- Any non-negative parent index is accepted in `process.parents.<n>.*`; out-of-range values render
  as `N/A`.
- If BoltConn only knows a parent PID, `.pid` is populated and `.name`, `.path`, and
  `.cmdline` render as `N/A`.
- Parent-chain fields depend on `dispatching.process_info_depth`. If the configured depth is too
  small, deeper parents render as `N/A`.

**Example emitted frame:**

```text
101:[14:23:01.456] curl <- bash -> example.com:443
```

## `.REQUEST`

Rule example:

```yaml
- .REQUEST:
    id: 200
    matches: PROCESS-NAME, untrusted-app
    timeout: 30
    request_route: CONTINUE
    fallback: REJECT
```

When the rule matches, BoltConn sends a request payload to subscriber `200` and waits for a
decision.

### Route Values

| Value | Used By | Behavior |
|---|---|---|
| `CONTINUE` | config and client | Continue evaluating later rules |
| `REJECT` | config and client | Reject the connection immediately |
| `BLACKHOLE` | config and client | Accept the connection and discard traffic |
| `FALLBACK` | client only | Apply the rule's configured `fallback` immediately |

### Server-to-Client Payload

`.REQUEST` uses the same `<id>:<message>` envelope, but `message` is JSON:

```text
200:{"sub_id":200,"request_id":42,"suggested_route":"CONTINUE","timeout":30,...}
```

Payload fields:

| Field | Description |
|---|---|
| `sub_id` | Subscriber ID from rule config |
| `request_id` | Monotonic request identifier generated by the server |
| `suggested_route` | Copy of `request_route` |
| `timeout` | Seconds before BoltConn applies `fallback` |
| `addr_src` | Source socket address |
| `addr_dst` | Original destination |
| `addr_resolved_dst` | Resolved destination, when available |
| `ip_local` | Local source IP, when available |
| `inbound_type` | `tun`, `http`, or `socks5` |
| `inbound_port` | Inbound port, when available |
| `inbound_user` | Inbound username, when available |
| `conn_type` | `tcp` or `udp` |
| `process_name` | Process name, when available |
| `process_cmdline` | Process command line, when available |
| `process_path` | Process executable path, when available |
| `process_cwd` | Process working directory, when available |
| `process_pid` | Process ID, when available |
| `process_tag` | BoltConn process tag, when available |
| `process_parent_all` | Array of collected parents |
| `time_hms_ms` | Local time formatted as `HH:MM:SS.mmm` |

Unavailable optional fields are omitted instead of serialized as `null`. `process_parent_all`
is always present and uses objects shaped like:

```json
{
  "pid": 999,
  "name": "bash",
  "path": "/bin/bash",
  "cmdline": "bash",
  "cwd": "/home/user"
}
```

If only a parent PID is known, the object still appears but `name`, `path`, `cmdline`, and
`cwd` are `null`.

### Client Response

The client replies on the same WebSocket:

```json
{
  "sub_id": 200,
  "request_id": 42,
  "route": "FALLBACK"
}
```

### Decision Flow

- No live subscriber for `id`: use `fallback` immediately
- Valid response before timeout: apply the requested route
- Client response `FALLBACK`: apply the configured fallback immediately
- Invalid `route`: use `fallback`
- Timeout or disconnect while waiting: use `fallback`

## `boltconn approve`

BoltConn ships an inline approval client for `.REQUEST`:

```bash
boltconn approve -i 31,32,100
```

Override mode skips config lookup entirely:

```bash
boltconn approve -i 31,32 --url 127.0.0.1:9001 --secret instrument-secret
```

**Arguments:**

| Argument | Description |
|---|---|
| `-i`, `--id` | Required comma-separated subscriber IDs |
| `--config` | Config directory used only when no override flag is present |
| `--url` | Instrument endpoint as `host:port`; using it skips config lookup |
| `--secret` | Secret override; using it also skips config lookup |

**UI behavior:**

- Runs until `Ctrl-C`
- Renders two panes plus a one-line footer on the normal terminal screen
- Left pane shows the pending queue, selected request, and remaining timeout
- Right pane shows details for the selected request
- `Left`/`Right` changes the selected request
- `Up`/`Down` scrolls the detail pane
- `Enter` or `a` approves by sending `CONTINUE`
- `x` denies by sending `FALLBACK`
- When a request's timeout elapses locally, the UI removes it without replying

## Minimal JavaScript Client

```javascript
const ws = new WebSocket(
  "ws://127.0.0.1:9001/subscribe?id=101,200&secret=instrument-secret"
);

ws.onmessage = (event) => {
  const sep = event.data.indexOf(":");
  const subId = Number(event.data.slice(0, sep));
  const body = event.data.slice(sep + 1);

  try {
    const payload = JSON.parse(body);
    if (payload.request_id !== undefined) {
      console.log("request", subId, payload);
      ws.send(
        JSON.stringify({
          sub_id: payload.sub_id,
          request_id: payload.request_id,
          route: "CONTINUE",
        })
      );
      return;
    }
  } catch (_) {}

  console.log("instrument", subId, body);
};
```

## Operational Notes

- `.INSTRUMENT` and `.REQUEST` share the same global subscriber ID space.
- Sharing an ID is intentional when one client should receive both formatted logs and interactive
  requests on a single WebSocket.
- Instrument publishing is best-effort. The internal ingress queue is bounded, and messages are
  dropped instead of blocking traffic handling when the queue is full.
